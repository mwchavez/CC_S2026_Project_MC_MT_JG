# Detection and Analytics Specification

## Overview

This document defines the detection logic used within the CloudHoney system. The goal of the detection layer is to analyze incoming events from the honeypot application, identify suspicious behavior patterns, and classify them based on predefined rules.

The system uses a rule-based approach to evaluate activity across simulated financial application endpoints. Events are processed in real time and evaluated for both single-event indicators and threshold-based patterns.

---

## Detection Pipeline Context

Detection occurs after raw events are captured and forwarded through the event pipeline:

```
Cloud Logging
    ↓
Pub/Sub (cloudhoney-events)
    ↓
Cloud Function (classify_event)
```

The `classify_event` function is responsible for:
- parsing incoming log data
- applying detection rules
- assigning severity levels
- storing processed results
- triggering alerts when applicable

---

## Event Structure

Each event processed by the detection engine is derived from structured logs generated by the honeypot.

Key fields include:
- `timestamp`
- `source_ip`
- `method`
- `path`
- `endpoint`
- `payload`
- `user_agent`
- `attack_context`

These fields are used as inputs for rule evaluation.

---

## Detection Strategy

CloudHoney uses two primary detection approaches:

### 1. Single-Event Detection

These rules trigger immediately when a specific pattern is identified within a single request.

Used for:
- SQL injection
- Wire fraud probing

---

### 2. Threshold-Based Detection

These rules evaluate behavior over time and trigger when activity exceeds a defined threshold.

Used for:
- Credential stuffing
- Account takeover reconnaissance
- Port scanning

Threshold logic is based on:
- number of attempts
- time window
- source IP consistency

---

## Detection Rules

### Rule 1: SQL Injection

**Type:** Single-event detection  
**Severity:** High  

**Logic:**
- Inspects payload data for known SQL injection patterns
- Examples include:
  - `OR 1=1`
  - `UNION SELECT`
  - `DROP TABLE`

**Trigger Condition:**
- Any request containing a matching pattern in the payload

**Purpose:**
Identifies attempts to manipulate backend queries through injected SQL commands.

---

### Rule 2: Wire Fraud Probe

**Type:** Single-event detection  
**Severity:** High  

**Logic:**
- Triggered when requests target:
  - `/transfer`
  - `/payment`
- Evaluates payload for suspicious routing or account values
  - repetitive digits
  - placeholder-style values (e.g., all zeros)

**Trigger Condition:**
- Suspicious payload patterns within financial transaction endpoints

**Purpose:**
Detects probing behavior targeting transaction functionality.

---

### Rule 3: Credential Stuffing

**Type:** Threshold-based detection  
**Severity:** High  

**Logic:**
- Monitors repeated POST requests to `/login`
- Tracks attempts from the same source IP

**Threshold:**
- More than 10 attempts within 60 seconds

**Trigger Condition:**
- Threshold exceeded

**Purpose:**
Identifies automated login attempts using multiple credential combinations.

---

### Rule 4: Account Takeover Reconnaissance

**Type:** Threshold-based detection  
**Severity:** Medium  

**Logic:**
- Monitors requests to `/account`
- Evaluates payload fields such as:
  - account identifiers
  - SSN lookups
  - session tokens

**Threshold:**
- More than 5 probing attempts within 60 seconds

**Trigger Condition:**
- Repeated probing behavior detected

**Purpose:**
Detects reconnaissance activity prior to account takeover attempts.

---

### Rule 5: Port Scan / Path Reconnaissance

**Type:** Threshold-based detection  
**Severity:** Medium  

**Logic:**
- Captures requests to unknown or unexpected endpoints
- Tracks number of distinct paths accessed by a single IP

**Threshold:**
- More than 8 distinct paths within 30 seconds

**Trigger Condition:**
- High volume of unique endpoint requests

**Purpose:**
Identifies scanning behavior used to map application surfaces.

---

## Severity Classification

Each detection rule is associated with a severity level:

- **High Severity**
  - SQL Injection
  - Wire Fraud Probe
  - Credential Stuffing

- **Medium Severity**
  - Account Takeover Reconnaissance
  - Port Scanning

Severity levels are assigned during classification and used for:
- alert prioritization
- dashboard visualization
- downstream analysis

---

## Event Processing Workflow

When an event is received:

1. Event is parsed from Pub/Sub message  
2. Relevant fields are extracted  
3. Detection rules are evaluated  
4. If a rule matches:
   - severity is assigned  
   - rule name is recorded  
5. Event is written to Firestore  
6. If conditions are met:
   - alert is published to Pub/Sub (`cloudhoney-alerts`)

---

## Output Data Structure

Processed events stored in Firestore include:

- `attack_type`
- `severity`
- `rule_matched`
- `source_ip`
- `timestamp`
- `path`
- `method`
- `payload`
- `user_agent`
- `alert_triggered`

This structure supports both analysis and visualization.

---

## Alert Triggering

Alerts are generated when:
- a single-event rule is triggered
- a threshold-based rule exceeds its defined limit

Alert messages include:
- severity level
- attack type
- source IP
- timestamp
- affected endpoint
- rule matched

These alerts are forwarded for notification processing.

---

## Summary

The CloudHoney detection engine uses a structured, rule-based approach to identify suspicious activity within a simulated financial environment.

The system combines:
- real-time event processing
- pattern-based detection
- threshold-based behavior analysis
- severity classification
- structured data output

This design enables accurate identification of common attack patterns while maintaining clear and explainable detection logic.
